使用的话很简单，固定写法如下：

多指触摸的话也就多了几个事件

使用与单指触摸大同小异

一些规律

  • Down 事件用于寻找谁来处理这次事件，然后之后的所有事件都会直接传递给它（期间仍会经过它的父 View），而不会再走 U字形
  • onInterceptTouchEvent() 只有一次返回 true 的机会，返回 true 后就不会再被调用
  • 父 View 通过 onInterceptTouchEvent() 返回 true 后，子 View 除了收到一次 Cancel 外，就再也不会收到任何事件

光听规律你可能会听不懂，接下来我把整个事件分发装换成与生活相关的东西

事件分发 -> 公司

层级

  • Activity：公司老总
  • 外层布局：资深程序员 A
  • 里层布局：中级程序员 B
  • 底层布局：入门程序员 C

方法

  • dispatchTouchEvent：接受到上级需求
  • onInterceptTouchEvent：是否自己想接手需求
  • onTouchEvent：自己处理需求
  • requestDisallowInterceptTouchEvent：签订了合同，防止被公司突然辞退

对于这个 requestDisallowInterceptTouchEvent 方法可以阻止上层布局调用 onInterceptTouchEvent 方法来拦截事件，看完下面整个流程你就能理解它的作用了

事件

  • Down：衡量需求难度，如果合适就可以接手（决定了需求谁来处理）
  • Move：上级询问进度（如果进度太慢，可以召回需求自己做）
  • Up：上级验收成果（这个时候你不一定就做完了需求，因为可能还有动画的执行）
  • Cancel：辞退事件（因为太菜而被公司辞退）

因为 Up 事件的调用栈与 Move事件的调用栈基本一致，所以后面不会单独写 Up 事件

requestDisallowInterceptTouchEvent 方法的注意事项

  • 这个方法虽然是签的合同，但不是终身合同，合同最长有效期只有这次需求结束，即下一次需求需要重新签订（下一次 Down 事件需要重新调用该方法）
  • 可以随时撕毁该合同，然后回归到程序员 A 每次都评审代码
  • 签订合同后是所有上级都不能评审代码了
  • 只能在签订人 dispatchTouchEvent 调用后才能签订，因为 dispatchTouchEvent 的 Down 事件会强制清空所有合同

灵活运用这个方法，你就可以解决一些常见的滑动冲突了

比如：之前我们提到 ScrollView 中有一个 Button，如果我们想实现只要手指触摸 Button 就不允许 ScrollView 滚动。直接在 Button 收到的 Down 事件调用 requestDisallowInterceptTouchEvent 即可

嵌套滑动

前面简单讲解了普通滑动的流程，如果你仔细思考会发现一个普通滑动有一个较严重的缺点：一旦父布局拦截了事件，子布局就再也不能收到事件（对应被辞退后总不能再召回来吧）

官方也意识到了这个问题，所以推出了嵌套滑动的解决方案

为了更好的兼容以前的 Android 版本，嵌套滑动最开始是以扩展包发布的，原理就是从底层 View 往上寻找需要嵌套滑动的 View，然后把事件传递给它

NestedScrollingChild2

嵌套滑动的事件发送者

虽然看着要实现的方法有很多，但其实官方专门写了一个帮助类 NestedScrollingChildHelper，你只需要在合适的位置调用上面的几个方法就可以了，具体怎么实现需要你们自己去看官方的源码了，因为一般情况下不会自己去写嵌套滑动的事件发送者，所以我就不讲了

这里给出一些文章：Android - 有趣的嵌套滑动

NestedScrollingParent2

嵌套滑动的事件接收者

与 NestedScrollingChild2 一样，官方也有一个帮助类 NestedScrollingParentHelper

NestedScrollingParent2 是嵌套滑动中用得最多的，理解了它每个方法的回调时机，你就能很快学会嵌套滑动了

这里面要注意以下几点：

  • 1版本的 NestedScrollingChild 和 NestedScrollingParent 因为设计的原因，只会收到非惯性事件，不能得到惯性事件
  • 2版本的嵌套滑动包含了惯性滑动，所以 onStartNestedScroll 会回调两次，并且惯性滑动的开始会在非惯性滑动的 onStopNestedScroll 前调用

NestedScrollView 源码简单分析

onInterceptTouchEvent

之前我们讲到这里是是否拦截事件的地方，我们先从 Down 事件开始看起

Move 是最重要的一部分，里面的逻辑实现了之前我们讲到的 Button 与 ScrollView 的问题

在分析 onIntercept 的 Move 前我们必须要提出两个前提条件：

  • 1、子 View 没有在 Down 中调用 requestDisallow 去禁止父布局拦截事件
  • 2、子 View 一定会在 Down 是拦截或处理事件，即子 View 的 onIntercept 或者 onTouchEvent 返回 true
  • 至于原因我们在后面进行分析
  • 这里补充说明：Button 这类控件就符合上面这两个条件

上面提到了这个 mTouchSlop，这是官方定义的“认为是滚动的最小移动距离”，除了 ScrollView 以外，RecyclerView 等一系列可以滚动的 View 都使用了它，它是一个常量（虽然是个常量，但它是根据不同的手机去调整的）

在 Move 的代码很简单，主要就是判断是否拦截，这里一定要记住 onIntercept 只有一次拦截的机会，拦截后就不会再被调用，所以后面的 Up 时间就没有必要继续看了

onTouchEvent

如果按照正常的思维，我们应该直接从 Down 事件开始分析，但之前我们提出了一个前提条件，所以 onTouchEvent 的 Down 事件是不会被调用的。

然后根据前面的前提条件，在 ScrollView 的移动距离超过了 mTouchSlop，onTouchEvent 就被调用，而因为前面是在 Move 中才拦截，则 onTouchEvent 只会收到 Move 后的事件

onTouchEvent 的 Move 就分析完了，后面就是 Up 事件

这就是 Button 在 ScrollView 中嵌套时的事件处理，我们再来看一下前提条件：

  • 1、子 View 没有在 Down 中调用 requestDisallow 去禁止父布局拦截事件
  • 2、子 View 一定会在 Down 是拦截或处理事件，即子 View 的 onIntercept 或者 onTouchEvent 返回 true

首先看第一个，这个得原因很简单，因为调用了 requestDisallow 后，父 View 的 onIntercept 就不会被调用

其他

onTouchEvent方法报黄

在重写 onTouchEvent 方法的时候会报黄，提示原因：

意思就是你重写了这个方法却没有重写和调用 performClick，为什么要重写和调用呢？

重写时可以不需要的，但如果不调用的话，你的 View 在 setOnClickListener 时会无效，查看 View 源码你就会知道，点击事件是在 onTouchEvent 的 Up 事件时调用，你重写了，如果不调用 super，设置的点击监听就失效了

但一般都自定义 View 了，还要什么官方的点击事件（一般都自己写方法向外提供），我的建议是直接加上注解取消报黄即可

VelocityTracker

速率追踪器，一个计算滑动速度的帮助类，官方源码中出现的很多

GestureDetector

手势识别器，整合了一系列的长按、滑动等操作，自定义黑书上有讲解

但这个一般是用于 View 中单独使用，算是一个处理滑动的工具，一般不用来解决滑动冲突

OverScroller

专门处理滑动距离和速度的一个工具类，如果你想实现惯性滑动等效果，可以试试它

ScrollView 和 RecyclerView 中的滑动距离都是依靠它来计算的

setOnTouchListener

设置触摸监听

作用于 View 中自带的 onTouchEvent 方法基本一致

但要注意的是他会先调用 OnTouchListener，如果你返回 true，就不会回调 onTouchEvent 了

自定义 Behavior

其实这个只要你理解了嵌套滑动，就能很快学会它

嵌套滑动一般需要自己去重写 View 才能使用，但使用 Behavior 就可以省去这个过程，协调者布局直接管理整个滑动事件分发，将你的 View 与 Behavior 进行绑定，间接实现了嵌套滑动的处理，达到解耦的效果

之前廖老师想实现 BottomSheetBehavior + Vp2 + Rv 的效果，但官方的 BottomSheetBehavior 有点坑，它只能跟第一个子布局进行嵌套滑动，导致官方的无法实现理想的效果

然后我写了一个自定义 BottomSheetBehavior：https://github.com/985892345/BottomSheetTest
